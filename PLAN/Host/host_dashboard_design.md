# AirBnB Clone Host Dashboard — Technical Design

## 1. Overview

**Architecture decisions:**

- **Pattern:** Modular Monolith — single repo, clear module boundaries between host, client, and admin
- **Frontend:** React.js (SPA) — component-based, React Router for navigation
- **Backend:** Flask (Python) — RESTful API, Blueprint-based module separation
- **Database:** Oracle Database + SQLAlchemy ORM — parameterized queries, schema migrations via Alembic (cx_Oracle dialect)
- **Auth:** JWT (access + refresh tokens) — bcrypt password hashing, 30-min access token, 7-day refresh token
- **State:** React Context + React Query — server state cached via React Query, UI state via Context
- **File storage:** Cloud storage (e.g., Cloudinary or S3) for property images and KYC documents — URLs stored in DB

**Data ownership principle:**

- Host Dashboard owns: host accounts, properties, bookings (source of truth), payments/wallets, reviews, messages, notifications, availability calendars
- Admin Dashboard accesses host data read-only via the `/api/admin/*` endpoints
- Client (Guest) interacts via the public `/api/rooms/*` and `/api/bookings/*` endpoints (Carl's client module)
- No shared `role` column — each module has its own auth tables

---

## 2. Database Schema (Oracle, 3NF)

All tables reside in a single Oracle schema (e.g., `AIRBNB_HOST`). The schema satisfies **Third Normal Form (3NF)**:
- Every non-key attribute depends on the whole primary key (no partial dependencies).
- Every non-key attribute depends only on the primary key, not on other non-key attributes (no transitive dependencies).
- All repeating groups are extracted into separate tables.

### 2.1 Auth & Host Profiles

**3NF notes:** `HOST_PROFILES` holds host-specific metadata separate from auth credentials (`HOSTS`). OTP codes are stored in a dedicated `OTP_VERIFICATIONS` table (no transitive dependency on host identity).

```sql
-- Host authentication credentials
CREATE TABLE HOSTS (
    id             NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    email          VARCHAR2(255) UNIQUE NOT NULL,
    password_hash  VARCHAR2(255) NOT NULL,
    status         VARCHAR2(20) DEFAULT 'awaiting_verification'
                       CONSTRAINT chk_host_status
                       CHECK (status IN ('awaiting_verification','active','suspended','inactive')),
    email_verified NUMBER(1) DEFAULT 0,
    created_at     TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at     TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- Host profile info (separate from auth — 3NF: no transitive dep on email)
CREATE TABLE HOST_PROFILES (
    id          NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    host_id     NUMBER UNIQUE NOT NULL REFERENCES HOSTS(id) ON DELETE CASCADE,
    full_name   VARCHAR2(100) NOT NULL,
    phone       VARCHAR2(20),
    avatar_url  VARCHAR2(500),
    bio         CLOB,
    is_superhost NUMBER(1) DEFAULT 0,
    created_at  TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at  TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- KYC documents (each document row is independent; no repeated groups)
CREATE TABLE HOST_KYC_DOCUMENTS (
    id              NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    host_id         NUMBER NOT NULL REFERENCES HOSTS(id) ON DELETE CASCADE,
    document_type   VARCHAR2(50)
                        CONSTRAINT chk_kyc_type
                        CHECK (document_type IN ('id_card','passport','drivers_license','selfie_with_id')),
    document_url    VARCHAR2(500) NOT NULL,
    status          VARCHAR2(20) DEFAULT 'pending'
                        CONSTRAINT chk_kyc_status
                        CHECK (status IN ('pending','approved','rejected')),
    review_notes    VARCHAR2(500),
    submitted_at    TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    reviewed_at     TIMESTAMP
);

-- OTP codes (separate table — not stored on HOSTS to avoid transitive dependency)
CREATE TABLE OTP_VERIFICATIONS (
    id          NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    email       VARCHAR2(255) NOT NULL,
    otp_code    VARCHAR2(6) NOT NULL,
    purpose     VARCHAR2(50)
                    CONSTRAINT chk_otp_purpose
                    CHECK (purpose IN ('email_verification','password_reset','withdrawal_confirm')),
    expires_at  TIMESTAMP NOT NULL,
    verified_at TIMESTAMP,
    created_at  TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- Payout accounts (one host may have many — repeated group extracted per 3NF)
CREATE TABLE PAYOUT_ACCOUNTS (
    id             NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    host_id        NUMBER NOT NULL REFERENCES HOSTS(id) ON DELETE CASCADE,
    account_type   VARCHAR2(20)
                       CONSTRAINT chk_payout_type
                       CHECK (account_type IN ('bank','gcash','maya')),
    account_name   VARCHAR2(100) NOT NULL,
    account_number VARCHAR2(50) NOT NULL,
    bank_name      VARCHAR2(100),
    is_default     NUMBER(1) DEFAULT 0,
    is_verified    NUMBER(1) DEFAULT 0,
    created_at     TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

### 2.2 Properties & Listings

**3NF notes:** Property amenities are in a junction table (`PROPERTY_AMENITIES`) rather than a comma-separated column. House rules, images, and pricing overrides are each in their own tables to eliminate repeating groups. `PROPERTY_IMAGES.display_order` depends only on `(property_id, display_order)` — no partial key deps.

```sql
-- Lookup table for amenity types (avoids repeating string literals)
CREATE TABLE AMENITIES (
    id    NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    name  VARCHAR2(100) UNIQUE NOT NULL,
    icon  VARCHAR2(50)
);

-- Core property record
CREATE TABLE PROPERTIES (
    id                 NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    host_id            NUMBER NOT NULL REFERENCES HOSTS(id) ON DELETE CASCADE,
    title              VARCHAR2(255) NOT NULL,
    description        CLOB,
    property_type      VARCHAR2(50)
                           CONSTRAINT chk_prop_type
                           CHECK (property_type IN ('entire_place','private_room','shared_room')),
    category           VARCHAR2(50),   -- house, apartment, villa, cabin, etc.
    status             VARCHAR2(30) DEFAULT 'pending_approval'
                           CONSTRAINT chk_prop_status
                           CHECK (status IN ('pending_approval','active','inactive','suspended','pending_review')),
    max_guests         NUMBER(3) NOT NULL,
    bedrooms           NUMBER(3) DEFAULT 0,
    beds               NUMBER(3) DEFAULT 0,
    bathrooms          NUMBER(3,1) DEFAULT 0,
    base_price         NUMBER(10,2) NOT NULL,
    cleaning_fee       NUMBER(10,2) DEFAULT 0,
    min_nights         NUMBER(3) DEFAULT 1,
    max_nights         NUMBER(3),
    cancellation_policy VARCHAR2(20) DEFAULT 'flexible'
                            CONSTRAINT chk_cancel_policy
                            CHECK (cancellation_policy IN ('flexible','moderate','strict')),
    instant_book       NUMBER(1) DEFAULT 0,
    created_at         TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at         TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- Property location (separate from PROPERTIES — location data is its own entity)
CREATE TABLE PROPERTY_LOCATIONS (
    id           NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    property_id  NUMBER UNIQUE NOT NULL REFERENCES PROPERTIES(id) ON DELETE CASCADE,
    street       VARCHAR2(255),
    city         VARCHAR2(100) NOT NULL,
    province     VARCHAR2(100),
    zip_code     VARCHAR2(20),
    country      VARCHAR2(100) DEFAULT 'Philippines',
    latitude     NUMBER(10,6),
    longitude    NUMBER(10,6)
);

-- House rules (separate table — rule_key/rule_value pairs avoid partial deps)
CREATE TABLE PROPERTY_RULES (
    id           NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    property_id  NUMBER NOT NULL REFERENCES PROPERTIES(id) ON DELETE CASCADE,
    rule_key     VARCHAR2(50) NOT NULL,  -- 'smoking','pets','parties','checkin_time','checkout_time'
    rule_value   VARCHAR2(255) NOT NULL,
    CONSTRAINT uq_property_rule UNIQUE (property_id, rule_key)
);

-- Junction table for property–amenity relationship (3NF: no many-valued attribute)
CREATE TABLE PROPERTY_AMENITIES (
    property_id  NUMBER NOT NULL REFERENCES PROPERTIES(id) ON DELETE CASCADE,
    amenity_id   NUMBER NOT NULL REFERENCES AMENITIES(id) ON DELETE CASCADE,
    CONSTRAINT pk_prop_amenity PRIMARY KEY (property_id, amenity_id)
);

-- Property images (display_order is a property of the image row, not of the property)
CREATE TABLE PROPERTY_IMAGES (
    id            NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    property_id   NUMBER NOT NULL REFERENCES PROPERTIES(id) ON DELETE CASCADE,
    image_url     VARCHAR2(500) NOT NULL,
    display_order NUMBER(3) DEFAULT 0,
    is_cover      NUMBER(1) DEFAULT 0,
    created_at    TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- Availability calendar (one row per date per property — no arrays/repeating groups)
CREATE TABLE AVAILABILITY_CALENDAR (
    id           NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    property_id  NUMBER NOT NULL REFERENCES PROPERTIES(id) ON DELETE CASCADE,
    date_value   DATE NOT NULL,
    status       VARCHAR2(20) DEFAULT 'available'
                     CONSTRAINT chk_avail_status
                     CHECK (status IN ('available','booked','blocked')),
    block_reason VARCHAR2(100),
    min_nights_override NUMBER(3),
    price_override      NUMBER(10,2),
    CONSTRAINT uq_property_date UNIQUE (property_id, date_value)
);
```

### 2.3 Bookings

**3NF notes:** `BOOKINGS` stores the booking record; price breakdown details that depend only on the booking are stored in `BOOKING_PRICE_DETAILS` to avoid a wide row with nullable columns. `BOOKING_CANCELLATIONS` captures cancellation data without making BOOKINGS transitively depend on cancellation reason.

```sql
-- Core booking record
CREATE TABLE BOOKINGS (
    id               NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    property_id      NUMBER NOT NULL REFERENCES PROPERTIES(id),
    host_id          NUMBER NOT NULL REFERENCES HOSTS(id),
    guest_external_id VARCHAR2(255) NOT NULL,  -- Guest's ID from client module
    guest_email      VARCHAR2(255) NOT NULL,
    check_in         DATE NOT NULL,
    check_out        DATE NOT NULL,
    guests_count     NUMBER(3) DEFAULT 1,
    status           VARCHAR2(20) DEFAULT 'pending'
                         CONSTRAINT chk_booking_status
                         CHECK (status IN ('pending','confirmed','in_progress','completed','cancelled','declined','disputed')),
    special_requests CLOB,
    created_at       TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at       TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- Price breakdown (depends fully on booking_id — no partial deps)
CREATE TABLE BOOKING_PRICE_DETAILS (
    id              NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    booking_id      NUMBER UNIQUE NOT NULL REFERENCES BOOKINGS(id) ON DELETE CASCADE,
    base_price      NUMBER(10,2) NOT NULL,   -- price per night at time of booking
    nights          NUMBER(5) NOT NULL,
    cleaning_fee    NUMBER(10,2) DEFAULT 0,
    service_fee     NUMBER(10,2) DEFAULT 0,
    commission_rate NUMBER(5,2) DEFAULT 0,
    commission_amount NUMBER(10,2) DEFAULT 0,
    total_price     NUMBER(10,2) NOT NULL,
    host_payout     NUMBER(10,2) NOT NULL
);

-- Decline and cancellation records (separate — not all bookings are cancelled)
CREATE TABLE BOOKING_CANCELLATIONS (
    id              NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    booking_id      NUMBER UNIQUE NOT NULL REFERENCES BOOKINGS(id) ON DELETE CASCADE,
    cancelled_by    VARCHAR2(10) CONSTRAINT chk_cancel_by CHECK (cancelled_by IN ('host','guest','admin')),
    reason          VARCHAR2(500),
    refund_amount   NUMBER(10,2) DEFAULT 0,
    cancelled_at    TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

### 2.4 Payments & Wallet

**3NF notes:** `WALLETS` has a 1:1 relationship with `HOSTS` (wallet balance is not a direct attribute of HOSTS, avoiding transitive dependency). `WALLET_TRANSACTIONS` records each atomic transaction. `WITHDRAWALS` is its own entity with a status lifecycle.

```sql
-- One wallet per host (1:1 — separated to avoid wide HOSTS table)
CREATE TABLE WALLETS (
    id                NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    host_id           NUMBER UNIQUE NOT NULL REFERENCES HOSTS(id) ON DELETE CASCADE,
    total_balance     NUMBER(12,2) DEFAULT 0,
    pending_balance   NUMBER(12,2) DEFAULT 0,
    available_balance NUMBER(12,2) DEFAULT 0,
    on_hold_balance   NUMBER(12,2) DEFAULT 0,
    updated_at        TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- Each wallet transaction is a separate row (no repeating groups)
CREATE TABLE WALLET_TRANSACTIONS (
    id               NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    wallet_id        NUMBER NOT NULL REFERENCES WALLETS(id),
    booking_id       NUMBER REFERENCES BOOKINGS(id),
    transaction_type VARCHAR2(30)
                         CONSTRAINT chk_txn_type
                         CHECK (transaction_type IN ('booking_payment','refund_deduction','withdrawal','admin_adjustment')),
    amount           NUMBER(12,2) NOT NULL,
    running_balance  NUMBER(12,2) NOT NULL,
    description      VARCHAR2(255),
    created_at       TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- Withdrawal requests (separate entity with its own lifecycle)
CREATE TABLE WITHDRAWALS (
    id                  NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    wallet_id           NUMBER NOT NULL REFERENCES WALLETS(id),
    payout_account_id   NUMBER NOT NULL REFERENCES PAYOUT_ACCOUNTS(id),
    amount              NUMBER(12,2) NOT NULL,
    fee                 NUMBER(10,2) DEFAULT 0,
    net_amount          NUMBER(12,2) NOT NULL,
    status              VARCHAR2(20) DEFAULT 'pending'
                            CONSTRAINT chk_withdrawal_status
                            CHECK (status IN ('pending','processing','processed','failed')),
    requested_at        TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    processed_at        TIMESTAMP,
    failure_reason      VARCHAR2(255)
);

-- PayMongo payment records (linked to booking — not duplicated on BOOKINGS)
CREATE TABLE PAYMENTS (
    id                   NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    booking_id           NUMBER NOT NULL REFERENCES BOOKINGS(id),
    paymongo_payment_id  VARCHAR2(255),
    amount               NUMBER(10,2) NOT NULL,
    payment_method       VARCHAR2(50),
    status               VARCHAR2(20) DEFAULT 'pending'
                             CONSTRAINT chk_payment_status
                             CHECK (status IN ('pending','paid','failed','refunded')),
    paid_at              TIMESTAMP,
    created_at           TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

### 2.5 Reviews

**3NF notes:** Sub-ratings (cleanliness, accuracy, etc.) are in `REVIEW_SUB_RATINGS` instead of 6 nullable columns on `REVIEWS`, eliminating repeating groups. `REVIEW_SUB_RATINGS.rating_type` + `review_id` form the composite key — each sub-rating depends on both.

```sql
-- Guest-to-property review
CREATE TABLE REVIEWS (
    id                  NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    booking_id          NUMBER UNIQUE NOT NULL REFERENCES BOOKINGS(id),
    property_id         NUMBER NOT NULL REFERENCES PROPERTIES(id),
    host_id             NUMBER NOT NULL REFERENCES HOSTS(id),
    reviewer_external_id VARCHAR2(255) NOT NULL,
    overall_rating      NUMBER(2,1) CONSTRAINT chk_rating CHECK (overall_rating BETWEEN 1 AND 5),
    review_text         CLOB,
    host_response       CLOB,
    host_response_at    TIMESTAMP,
    is_visible          NUMBER(1) DEFAULT 1,
    created_at          TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- Sub-ratings extracted to avoid repeating groups (3NF)
CREATE TABLE REVIEW_SUB_RATINGS (
    id          NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    review_id   NUMBER NOT NULL REFERENCES REVIEWS(id) ON DELETE CASCADE,
    rating_type VARCHAR2(30)
                    CONSTRAINT chk_sub_rating_type
                    CHECK (rating_type IN ('cleanliness','accuracy','communication','location','checkin','value')),
    rating      NUMBER(2,1) CONSTRAINT chk_sub_rating CHECK (rating BETWEEN 1 AND 5),
    CONSTRAINT uq_review_sub_rating UNIQUE (review_id, rating_type)
);

-- Host-to-guest review (separate entity — different reviewer direction)
CREATE TABLE GUEST_REVIEWS (
    id                  NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    booking_id          NUMBER UNIQUE NOT NULL REFERENCES BOOKINGS(id),
    guest_external_id   VARCHAR2(255) NOT NULL,
    host_id             NUMBER NOT NULL REFERENCES HOSTS(id),
    cleanliness_rating  NUMBER(2,1),
    communication_rating NUMBER(2,1),
    house_rules_rating  NUMBER(2,1),
    overall_rating      NUMBER(2,1) CONSTRAINT chk_guest_rating CHECK (overall_rating BETWEEN 1 AND 5),
    review_text         CLOB,
    created_at          TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

### 2.6 Messaging

**3NF notes:** `CONVERSATIONS` are keyed by `(booking_id, host_id, guest_external_id)` as a composite identity, but stored with a surrogate key for FK references. `MESSAGES` depends fully on `conversation_id` — no partial or transitive deps.

```sql
-- One conversation per booking (unique constraint enforces this)
CREATE TABLE CONVERSATIONS (
    id                  NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    booking_id          NUMBER NOT NULL REFERENCES BOOKINGS(id),
    host_id             NUMBER NOT NULL REFERENCES HOSTS(id),
    guest_external_id   VARCHAR2(255) NOT NULL,
    last_message_at     TIMESTAMP,
    host_unread_count   NUMBER(5) DEFAULT 0,
    guest_unread_count  NUMBER(5) DEFAULT 0,
    created_at          TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    CONSTRAINT uq_conversation_booking UNIQUE (booking_id)
);

-- Individual messages (depend fully on conversation_id)
CREATE TABLE MESSAGES (
    id               NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    conversation_id  NUMBER NOT NULL REFERENCES CONVERSATIONS(id) ON DELETE CASCADE,
    sender_type      VARCHAR2(10) CONSTRAINT chk_sender_type CHECK (sender_type IN ('host','guest')),
    sender_id        VARCHAR2(255) NOT NULL,
    content          CLOB,
    attachment_url   VARCHAR2(500),
    is_read          NUMBER(1) DEFAULT 0,
    sent_at          TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- Saved quick reply templates (belong to host — no transitive dep on message content)
CREATE TABLE QUICK_REPLY_TEMPLATES (
    id         NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    host_id    NUMBER NOT NULL REFERENCES HOSTS(id) ON DELETE CASCADE,
    label      VARCHAR2(100) NOT NULL,
    content    CLOB NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

## 4. Admin API Contract (Exposed by Host Module)

Admin Dashboard calls these endpoints. API key stored in admin's `system_settings` table under keys `carl_api_base_url` and `carl_api_key`.

**Authentication:** `X-API-Key` header.

**Listing endpoints**
- `GET /api/admin/listings?status=pending` — listings pending approval (title, host email, photos, price, location)
- `POST /api/admin/listings/:id/approve` — set property status = active
- `POST /api/admin/listings/:id/reject` — set status = inactive, store reason in `PROPERTY_RULES` or response body
- `POST /api/admin/listings/:id/suspend` — set status = suspended

**Host endpoints**
- `GET /api/admin/hosts/:external_id` — host profile, verification status, badge, total listings count

**Guest endpoints**
- `GET /api/admin/guests/:external_id` — guest summary (name, email, total bookings) pulled from booking records

**Booking endpoints**
- `GET /api/admin/bookings` — all bookings (paginated, filterable by status/date)
- `GET /api/admin/bookings/:id` — booking detail with price breakdown

## 6. Data Flow (High Level)

```
[Guest / Client Module]
        ↓ books via public API
[Flask API — Host Module]
        ↓
[Oracle Database — AIRBNB_HOST schema]
        ↓ /api/admin/* endpoints
[Admin Dashboard — Streamlit]
        ↓ approval / moderation actions
[Flask API — Host Module]  ←→  [PayMongo — payment webhooks]
```

**Sync notes:**
- Admin reads listing/host/guest data from Host module's read-only API endpoints.
- PayMongo sends payment webhooks to `POST /api/webhooks/paymongo` → updates `PAYMENTS` and `WALLETS`.
- No real-time sockets initially — React Query polls for updates every 30 seconds on active pages.

## 7. Security & Auth

- **Authentication:** JWT (access token 30 min, refresh token 7 days stored in HttpOnly cookie)
- **Password hashing:** bcrypt (cost factor 12)
- **Rate limiting:** 5 failed login attempts per 15 min per IP (tracked in a Redis counter or DB fallback)
- **Audit trail:** every host write action logged to `AUDIT_LOG`
- **API key for admin:** issued per-environment, stored in `system_settings` on admin side
- **KYC documents:** stored in cloud storage; URLs in DB; access requires signed URL with short expiry
- **CORS:** restricted to known frontend origins

**Per-module auth isolation:**
- Host auth uses `HOSTS` table — no shared role column with admin or guest
- Guest auth is handled by client module (Carl's separate schema/app)
- Admin auth uses `admin_accounts` table in admin's PostgreSQL DB
